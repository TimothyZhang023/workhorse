# 🎯 Web Agent Assistant (Cowhouse) 测试规范 (Test Spec)

这份文档用于指引在使用 Cowhouse (CW) 网页端 AI 助手时，如何完整进行各项核心功能的前后端端到端测试。

## 1. 基础配置测试 (Base Configuration)

### 1.1 端点与模型设置

- **前置条件**：准备一个可用的 OpenAI 兼容 API。
- **测试步骤**：
  1. 通过左侧导航栏或者主界面进入 **“设置 (Settings)”** -> **“模型服务商”**。
  2. 添加或编辑服务商，配置信息如下：
     - **API Base URL**: `http://192.168.3.242:1234/v1`
     - **API Key**: 输入对应鉴权 Token（随意填写符合约束的字串）。
  3. 点击 **“同步当前通道模型”**（或同等刷新按钮），确保界面提示成功。
  4. 检查可用模型列表，应当包含并可选择：`qwen3.5-0.8b` (或类似的已知可用短模型)。
- **预期结果**：端点配置保存成功，模型列表正确渲染并可设为默认。

---

## 2. MCP 能力与插件验证 (MCP & Plugins)

### 2.1 添加自定义 MCP 服务

- **测试步骤**：
  1. 进入 **“设置”** -> **“MCP 插件”**。
  2. 点击添加服务，测试以下类型之一：
     - `sse` 类型服务，填入带有自定义 `Headers` 规范的 URL 和 `{"Authorization": "Bearer test"}`。
     - `stdio` 类型服务，填入安全的本地执行指令串。
  3. 保存。
- **预期结果**：McpManager 正常启动连接，系统不发生崩溃，成功拉取到子服务注册的 `Tools`。

---

## 3. 技能库测试 (Skill Library)

### 3.1 创建与编辑技能

- **测试步骤**：
  1. 进入 **“技能库 (Skills)”**。
  2. 点击 **“添加技能”**。
  3. 填写 `技能名称` (如 "数据搜索专家")，`系统提示增强` (如 "你必须使用特定的 JSON 格式返回...")。
  4. 保存此技能，然后再次点击“编辑”检查数据回调。
- **预期结果**：保存顺畅且界面能即时刷新；打开弹窗没有卡顿冻结问题，且内容能正确回显。

---

## 4. 任务编排测试 (Agent Tasks)

### 4.1 编排与启动 Agent 实例

- **测试步骤**：
  1. 进入 **“任务编排 (Agent Tasks)”**。
  2. 点击 **“创建任务”**。
  3. 关联上述第 3 步创建的 **技能**。
  4. 关联特定的 **MCP 工具** (若环境允许)。
  5. 填写任务的核心 System Prompt。
  6. 在列表中点击该任务的 **“启动”** 按钮。
- **预期结果**：控制台不会输出 SQL stringify related Error，任务可正常被分配唯一的 Conversation ID 并跳转或在后台正常运转。执行完成后，在聊天窗口可以看到带有此任务标记的会话记录。

---

## 5. 对话功能与降级测试 (Chat Interaction)

### 5.1 基础模型对话

- **测试步骤**：
  1. 进入 **“对话 (Chat)”** 界面。
  2. 新建对话，保证右上角所选模型为刚才测试的 `qwen3.5-0.8b`。
  3. 发送简单测试句：`"你好，请问 1+1 等于几？"`
- **预期结果**：前端展示正常的 Typing UI，随后流式返回（Stream SSE）结果且控制台没有异常。
- **降级容错**：尝试关闭 `http://192.168.3.242:1234` 后段服务重新请求，UI 应正确捕捉到 `Fetch failed` 或类似网络错误，并出现 **“重新生成”** 等允许容错的机制。

---

## 6. 定时调度测试 (Cron Jobs)

### 6.1 Cron 表达式周期测试

- **测试步骤**：
  1. 进入 **“定时调度 (Cron Jobs)”** 页面。
  2. 点击 **“创建调度”**。
  3. 将运行任务指向刚才创建的某一项 **“任务 (AgentTask)”**。
  4. 输入测试用的 Cron 表达式 (如快速检查：`*/2 * * * *` 即每 2 分钟触发一次)。
  5. 勾选 **启用**，观察后台终端日志（Node.js 端）。
- **预期结果**：Node 端 `cronRunner` 能够成功注册事件。两分钟后日志中应该出现 `[CronRunner] Executing scheduled task` 的成功回显，且面板中“上次执行”状态从空变为 `success`。
